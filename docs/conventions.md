# Patterns

Project-specific conventions for Savyy. Where clean-code.md covers universal principles, this file covers how we do things in this codebase specifically.

---

## Naming

### Verb families
Pick one verb per operation and never mix. Applies across store mutations, component handlers, and UI copy.

| Operation | Use | Never mix with |
|---|---|---|
| Adding items ‚Äî store/data layer | `add` | `create`, `new`, `insert` |
| Adding items ‚Äî UI (buttons, titles) | `New [item]` | `Add new`, `Create` |
| Adding items ‚Äî UI (empty states) | `Create new [item]` | `Add your first` |
| Removing ‚Äî store | `delete` | `remove`, `destroy`, `drop` |
| Removing ‚Äî UI | `delete` | `remove`, `destroy` |
| Changing ‚Äî store | `update` | `modify`, `set` (except observable `.set()`) |
| Changing ‚Äî UI | `edit` | `modify`, `change` |
| Synchronous reads | `get` | `read`, `retrieve` |
| Async data loading | `fetch` | `load`, `get`, `pull` |
| Navigation | `navigate` | `go`, `open`, `redirect`, `push` |
| Visibility | `show` / `hide` | `reveal`/`collapse`, `display` |
| Overlay open state | `open` / `close` | `show`/`hide`, `isOpen` |
| Selection | `select` / `deselect` | `pick`, `choose`, `check` |
| Initialization | `init` | `initialize`, `setup`, `bootstrap` |
| Resetting state | `reset` | `clear`, `empty`, `flush` |
| Data ‚Üí display string | `format` | `render`, `display`, `stringify` |
| String ‚Üí data | `parse` | `decode`, `deserialize` |
| Shape A ‚Üí shape B | `map` | `transform`, `convert`, `serialize` |
| Validation | `validate` | `check`, `verify` |
| Async success callback | `onSuccess` | `onComplete`, `onDone` |
| Async error callback | `onError` | `onFailure`, `onFail` |
| Counting items | `count` | `length`, `size`, `total`, `num` |
| Sum of values | `total` | `sum`, `aggregate` |

### Event handlers
`on` for props (the interface), `handle` for implementations.

```tsx
// ‚úÖ
<TransactionRow onDelete={handleDelete} onChange={handleChange} />

function handleDelete(id: string) { ... }
function handleChange(value: string) { ... }
```

### Predicate naming

| Prefix | Meaning | Example |
|---|---|---|
| `is` | boolean state | `isSelected`, `isLoading` |
| `has` | existence check | `hasLabels`, `hasError` |
| `can` | permission / capability | `canDelete`, `canEdit` |

### Collection naming
Plural is enough ‚Äî no redundant suffixes.

```ts
// ‚úÖ transactions, labels, selectedIds
// ‚ùå transactionList, labelArray, transactionCollection
```

### ID naming
Plain `id` for the entity's own identifier. Qualified `XId` for foreign references.

```ts
interface Transaction {
  id: string          // own identity
  labelIds: string[]  // foreign reference
}
```

### Observable naming
Always end with `$`.

```ts
const transactions$ = observable([])     // observable
const transactions = transactions$.get() // plain value
```

### Component suffix
Use `Row` for items inside a table or list layout: `TransactionRow`, `LabelRow`.

### Avoid negative booleans

```ts
// ‚úÖ isEnabled, isVisible
// ‚ùå isDisabled, isHidden, isNotValid
```

## Folder conventions

### `store/` ‚Äî state only
One file per domain. Each file owns its observable, computed, mutations, types, and mappers. No React imports. See `docs/architecture.md` for the full store design pattern.

### `database/` ‚Äî generated, do not edit
`database.types.ts` is auto-generated by `supabase gen types typescript`. Re-run it after any schema change. `supabase-client.ts` is the typed singleton ‚Äî import from here everywhere.

### `lib/` ‚Äî product-agnostic utilities only
Code you could copy-paste into any project unchanged. No product concepts, no store imports, no feature-specific logic.

```ts
// ‚úÖ lib/ ‚Äî works in any project
app/lib/utils.ts              // cn(), generateId()
app/lib/use-keyboard-shortcut.ts

// ‚ùå lib/ ‚Äî product-specific, belongs in the feature folder
app/lib/label-colors.ts       // ‚Üí app/components/labels/
```

### `components/ui/` ‚Äî hands off
Generated by shadcn. Never edit directly.

### `components/[feature]/` ‚Äî can access store
Feature components read from computed observables and call store mutations directly. No prop drilling for data.

### `routes/` ‚Äî thin
Route files own the page layout and document title only. No business logic ‚Äî that lives in components and the store.

## File imports

Use path aliases everywhere. Never relative `../../../`.

```ts
// ‚úÖ
import { cn } from "~/lib/utils"
import { transactions$ } from "~/store/transactions"

// ‚ùå
import { cn } from "../../lib/utils"
```

Import order:
1. React
2. Third-party libraries
3. Internal `~/` imports
4. Relative imports (same directory only)
5. Type imports

## DB ‚Üî domain type mapping

Supabase types use snake_case and reflect the raw DB schema. Domain types use camelCase and are the shape the rest of the app works with. These are always different ‚Äî never use DB types directly in components.

| DB (Supabase / snake_case) | Domain (app / camelCase) |
|---|---|
| `created_at` | `createdAt` |
| `user_id` | (stripped ‚Äî never exposed to components) |
| `label_ids` | `labelIds` |
| `snake_case_field` | `camelCaseField` |

Mappers live in the store file alongside the types they serve. Two mappers per domain:
- `mapXRowToX(row: XRow): X` ‚Äî DB ‚Üí domain (used in the computed)
- `mapXToXRow(x: X, userId: string): XRow` ‚Äî domain ‚Üí DB (used in mutations)

```ts
// ‚úÖ mapper in store/labels.ts
export function mapLabelRowToLabel(row: LabelRow): Label {
  return {
    id: row.id,
    name: row.name,
    color: row.color,
    createdAt: row.created_at,
  }
}

export function mapLabelToLabelRow(label: Label, userId: string): LabelRow {
  return {
    id: label.id,
    name: label.name,
    color: label.color,
    created_at: label.createdAt,
    user_id: userId,
  }
}
```

`user_id` is injected in mutations (fetched from `supabaseClient.auth.getUser()`). It is never part of the domain type ‚Äî components have no reason to know about it.

## Store pattern

Types, mappers, observable, and mutations all live in the same store file.

```ts
// store/labels.ts

// 1. Domain type (camelCase, app-facing)
export interface Label { id: string; name: string; color: string; createdAt: string }

// 2. Mapper functions (DB row ‚Üî domain type)
export function mapLabelRowToLabel(row: LabelRow): Label { ... }
export function mapLabelToLabelRow(label: Label, userId: string): LabelRow { ... }

// 3. Private store + public computed
const labelsStore = observable({ items: syncedSupabase({ ... }) })
export const labels$ = computed(() => labelsStore.items.get().map(mapLabelRowToLabel))

// 4. Mutations
export async function addLabel(label: Pick<Label, "name" | "color">) { ... }
export async function updateLabel(id: string, patch: Partial<Pick<Label, "name" | "color">>) { ... }
export function deleteLabel(id: string) { ... }
```

The store is private. The computed (`labels$`) is the public interface for reads. Mutations are the public interface for writes.

### Reading observables in components

```tsx
// ‚úÖ
const labels = useValue(labels$)

// ‚úÖ with inline computation
const expenseTotal = useValue(() =>
  transactions$.get().filter(t => t.amount < 0).reduce((sum, t) => sum + t.amount, 0)
)

// ‚ùå not reactive
const labels = labels$.get()
```

### Writing to observables
Always via store mutation functions, never directly from components.

```ts
// ‚úÖ
addTransaction(formData)

// ‚ùå
transactionsStore.items[0].set(...)
```

## Data conventions

- **IDs**: UUID v4, generated client-side with `generateId()` from `~/lib/utils`
- **Amounts**: stored as integers in cents (`‚Ç¨42.50` ‚Üí `4250`). Negative = expense, positive = income.
- **Dates**: ISO 8601 strings (`"2024-03-15"`). All manipulation via `date-fns`.
- **Currency**: stored per-transaction (`"EUR"` | `"USD"`). User has a global default setting.

## Styling

- Tailwind utility classes only ‚Äî no inline `style={{}}` except for truly dynamic values
- Use `cn()` for conditional classes
- CSS variables defined in `app.css`, consumed by Tailwind
- Dark mode via `.dark` class on `<html>`

```tsx
// ‚úÖ
<div className={cn("rounded-lg border p-4", isSelected && "bg-accent border-primary")}>
```

## Keyboard shortcuts

All shortcuts go through `useKeyboardShortcut`. Never attach raw `keydown` listeners in components.

```ts
// ‚úÖ
useKeyboardShortcut("cmd+k", () => openCommandPalette())

// ‚ùå
window.addEventListener("keydown", (e) => { ... })
```


- No `useEffect` for data synchronization (use Legend State reactivity)
- No `useState` for data that belongs in the store
- No generic variable names (`data`, `result`, `item`, `temp`, `value`)
